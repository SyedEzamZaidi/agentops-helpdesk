## Architectural insights logged today
1. **Connector body schema is import-time, not API-derived** — missing fields in spec = invisible to LLM downstream
2. **Tool inputs run as Topic.Input.\*** — design model (tool) ≠ execution model (ephemeral topic)
3. **Connector defaults apply only when param is hidden from LLM** — exposure = LLM authority
4. **Completion mode by output cardinality** — single + critical → Specific Response. List or read-only → Generative
5. **Origin attribution is write-side only** — GET ops don't need u_agent_name; POST/PATCH do
6. **Four identity layers per ticket** — connection (maker), business (caller_id), operating (opened_by), origin (u_agent_name). Maker-provided solves 1 of 4
7. **Channel auth ≠ target system identity** — Teams injects email, ServiceNow indexes by sys_id, translation is your responsibility

## ADRs added today
- ADR-007: Maker-provided credentials as tactical bridge between End user and Client Credentials grants
- ADR-008: caller_id resolution architecture (Pattern 1 — 5th connector action GetUserBySysProperty)

## Decisions
- Maker-provided over End user credentials on all 4 tools — closer approximation to ADR-005 production target without needing Zurich Entity Profile fix
- "Only when referenced by topics or agents" on all 4 tools — Triage owns routing decision, no tool self-triggers
- "Ask user before running" = Yes for POST/PATCH (HITL gate), No for GET (read-only)
- Hardcoded `u_agent_name = "AgentOps ServiceNow"` on Create + Update — constant value, no LLM decision needed
- 5 of 11 Update body fields NOT exposed as tool inputs (re-classification rare, defer to IT review)
- `sysparm_fields` + `sysparm_display_value` NOT exposed to LLM — locked at connector default for response shape predictability
