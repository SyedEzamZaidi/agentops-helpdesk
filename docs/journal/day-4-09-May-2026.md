### Phase 2 — Credentials hygiene
Spent more time on this than expected. Worth it.

Mentor walked through the credential layering pattern:

| Credential | Lives in |
|---|---|
| Admin password (human login) | Password manager |
| OAuth Client ID | Power Platform Environment Variable (non-secret) |
| OAuth Client Secret | Azure Key Vault → referenced by EV (Secret type) |

This is the pattern AB-620 and PL-400 will quiz on. The wrong answer they bait you with is "store in connector definition." The right answer is always "Key Vault, referenced via EV." Logged this for exam prep.

Also got the OAuth concept properly. Two-step exchange: client secret → access token → API call. The secret is the long-lived credential (rotated quarterly or on leak). The token is the short-lived disposable thing that flies across the network on every API call. Stolen token expires in 30 minutes. Stolen secret is scoped to whatever the registered app is allowed to do — not full god mode like a stolen admin password.

Authorization hierarchy: Admin password > Client secret > OAuth token. Each layer can mint the level below it, never the level above. Principle of least privilege applied to credential design.

Pushed back hard on this one — initially asked "if a hacker has the secret, can they reconfigure the app to give themselves more permissions?" The answer is no — admin operations require admin auth, separately. OAuth separates *operating the system* from *configuring the system*. Compromising one does not compromise the other. That is the whole point.

### Phase 4 — REST API Explorer
This is the page our entire ServiceNow Agent will be built on.

Made the first live API call:
```
GET https://dev306311.service-now.com/api/now/table/incident?sysparm_limit=1
```

Got back a JSON response with one full incident record. ~80 fields. Mentor decoded which 10 actually matter for our agent (number, sys_id, short_description, description, state, priority, category, caller_id, assigned_to, comments, work_notes, opened_at, resolved_at).

Two ServiceNow concepts hit me:

**number vs sys_id** — every record has both. number is human-readable (INC0000060), sys_id is a 32-char internal hash. Users quote number; APIs use sys_id. Agent has to translate one to the other, which is a two-step lookup pattern.

**Reference fields** — fields like caller_id come back as `{ link, value }` objects pointing to another table (sys_user). The raw API response is full of these opaque sys_id pointers. Adding `sysparm_display_value=true` resolves them inline to human-readable values. Almost always want this for our agent so the LLM gets readable strings, not hashes.

State codes also worth memorizing: 1=New, 2=In Progress, 3=On Hold, 6=Resolved, 7=Closed, 8=Canceled. Default install. Customizations leave gaps.

## What I learned (architectural)
The credential layering pattern (Admin password > Client secret > OAuth token) is not just exam content — it is the design principle for the whole project. Every external integration we add (ServiceNow, Azure Key Vault, App Insights, GitHub Actions in Week 4) reuses this same three-layer pattern. Get it right once, repeat it everywhere.

ServiceNow's REST Table API is one universal pattern (`/api/now/table/{tableName}`) for CRUD on any table in the system. The agent will only ever use four operations: GET (collection with filter), GET (single by sys_id), POST (create), PATCH (update). Memorizing those four is sufficient — everything else is wrapping.

`sysparm_display_value=true` is the difference between an LLM seeing "Joe Employee" or seeing "681ccaf9c0a8016400b98a06818d57c7". Always want the former. AB-620 will likely test this.
