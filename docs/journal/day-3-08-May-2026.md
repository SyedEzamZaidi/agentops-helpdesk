## What actually happened
None of the planned setup work. The session got pulled into eval forensics and three architectural decisions that turned out to be more important than the original plan.

The day broke into four phases.

### Phase 1 — System topic audit
Started with the question "should we audit all 9 system topics now or defer to Week 2?" Initial mentor recommendation was defer. Pushed back: ADR-001's Escalate finding had already proven that auto-created topics override prompts, so deferring meant running 8 unaudited interrupt paths during the most active prompt-iteration phase. Mentor agreed and reversed.

Did a 20-minute defensive smoke audit of all 9 system topics. No new landmines found beyond Escalate. Confirmed Conversational boosting is the Knowledge-tab-driven Generative answers pipeline (NOT generative orchestration) — this matters for the Knowledge Agent RAG strategy in Week 3. Captured findings in ADR-002.

Decision logged: Escalate stays disabled. Other 8 topics keep default config with explicit re-visit triggers (when ServiceNow auth lands, when App Insights wires up, when persona pass happens before LinkedIn pivot).

### Phase 2 — Eval iteration 1 (the trap)
Ran the Day 2 test set against the agent with Escalate disabled. Result: 4 pass / 6 fail. Looked like progress over Day 2 baseline (which had 6/1/3).

Read the responses carefully. Four "passes" had full IT helpdesk guides with citations to support.oakland.edu and masonadvisory.com. The Triage Agent has zero Knowledge sources by design. Those answers were not coming from the prompt.

This was the architectural failure mode — the agent was not classifying or routing, it was hallucinating helpdesk answers from the public web with cited URLs. Recruiters reading this in a portfolio would catch it instantly.

Traced to agent-level Web Search being enabled. Disabled it. Captured in ADR-004.

### Phase 3 — Eval iteration 1.5 (the deeper trap)
With Web Search off, re-ran the eval. Result: 0/10. Every response was the Fallback topic's default message ("I'm sorry, I'm not sure how to help with that").

This was not a regression. It was the first uncontaminated signal we had ever gotten from this agent.

Diagnosed: "Allow ungrounded responses" had been flipped OFF based on Day 2 advice. For a router agent with no Knowledge sources, that setting blocks the only legal response path. The Day 2 advice was wrong — flipped from a single-agent mental model to a multi-agent system without re-deriving the implications.

The correct rule: "Allow ungrounded responses" is per-agent, not global. Routers need ON. Grounded specialists need OFF. Captured in ADR-003.

Mentor acknowledged the Day 2 mistake directly. Good. I do not want sycophancy here.

### Phase 4 — Eval iteration 2 (the clean signal)
Web Search OFF, Allow ungrounded responses ON, Escalate disabled.

Result: 8 pass / 2 fail by auto-grader. 10/10 by architectural review.

Both grader "fails" were correct routing behavior — the agent classified how-to questions and routed to the (not-yet-built) knowledge agent. The auto-grader marked these as Fail because its rubric is "did the response answer the user's question?" For a router, the correct behavior — refuse to answer, hand off — looks like failure to a generic helpfulness rubric.

This is the architectural justification for building a custom Foundry eval harness in Week 3 with role-aware metrics. Promoted from "nice-to-have" to "necessity." Captured in iteration-2 eval doc.

## Findings (CV-relevant)
1. Web Search at agent level overrides routing prompts. Discovered via eval forensics.
2. "Allow ungrounded responses" is a per-agent setting determined by architectural role, not a global safety toggle.
3. Auto-graders are role-blind. Built-in evals cannot grade multi-agent routing behavior — custom harnesses needed.
4. Conversational boosting is Knowledge-tab-driven RAG, not generative orchestration. Affects Knowledge Agent design.
5. Azure RAI content filters are uniformly applied across contexts. Same filter for production user inference and internal test-case generation. No per-context tuning in Copilot Studio. This is partial justification for BYO-Foundry-model with per-deployment custom filters in Week 3.

## What I learned (architectural)
The Day 2 baseline (6 pass / 1 fail / 3 errors) was not a real signal. Web Search was masking everything — passes were not the prompt working. Today's eval is the first run where the prompt is actually in the response path with no masking layers.

Three knobs together define the response surface of any Copilot Studio agent in this project:
- Generative AI orchestration (ON for all)
- Allow ungrounded responses (per-agent: ON for routers, OFF for grounded specialists)
- Web Search (OFF for all routers and specialists; deferred decision for Knowledge Agent)

Any new agent's spec must declare values for all three explicitly. This is now the playbook.

## What I learned (process)
Eval-driven iteration works when you read responses carefully, not just the pass/fail count. The 4/10 result on iteration 1 looked like progress and was actually the worst kind of failure — silent architectural rot disguised as success. The 0/10 result on iteration 1.5 looked like regression and was actually the first clean signal.

Trust the eval, but read what's underneath the score.
