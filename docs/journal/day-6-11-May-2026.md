## Wiring blocker discovered
Microsoft's pre-built ServiceNow connector does NOT support Client Credentials grant. Authentication Type dropdown offers:
- Use OAuth2 (Authorization Code under the hood — opens user login browser)
- Basic Authentication (deprecated)
- Microsoft Entra ID OAuth using Certificate
- Microsoft Entra ID User Login

None map cleanly to the Client Credentials architecture we designed.

## Four real paths forward (decision tomorrow)
1. Authorization Code compromise — fastest, breaks service-account design
2. Microsoft Entra ID Certificate — closer to design, 2-3 hrs new work, needs Entra app + cert
3. Custom connector with Client Credentials — pulls Week 3 work forward, 1-2 days
4. ServiceNow MCP server — modern AI-native protocol, learning curve unknown

## Stopping point
Architecturally correct call to stop here. Decision deserves fresh-head investigation, not fatigue-driven pick. Tomorrow morning: 30 min on MCP docs, 30 min on Entra Certificate docs, then decide.

## Learnings
- Premium connectors have licensing implications worth knowing for AB-620 / PL-400
- Power Platform OAuth callback URL is universal: `https://global.consent.azure-apim.net/redirect`
- ServiceNow's classic OAuth Application Registry UI exposes fields the new UI hides
- Trusted subsystem pattern: service account has broad permissions, agent enforces per-user authorization in code/prompt
